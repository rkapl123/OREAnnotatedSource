76/*! Class holding the information on the fixings required to price a trade (or a portfolio of trades). */
77class RequiredFixings {
78public:
79 /*! Gives back the dates for which fixings will be required to price the trade assuming a given \p settlementDate.
80 If the \p settlementDate is not provided or is set equal to \c QuantLib::Date(), the settlement date in the
81 implementation is assumed to be the \c Settings::instance().evaluationDate().
82
83 If a cashflow payment is deemed to have already occurred relative to the settlement date, then no fixing is
84 needed. The determination of whether a cashflow has or has not occurred will in general rely on a call to \c
85 CashFlow::hasOccurred which is important in cases where the cash flow payment date falls on the settlement date.
86
87 Another important case is where a cash flow fixing date occurs on the settlement date. In this case, we should
88 always add the fixing date to the set of fixing dates regardless of
89 \c Settings::instance().enforcesTodaysHistoricFixings(). */
90 std::map<std::string, std::set<QuantLib::Date>>
91 fixingDatesIndices(const QuantLib::Date& settlementDate = QuantLib::Date()) const;
92
93 /*! Adds a single fixing date \p fixingDate for an index given by its ORE index name \p indexName arising from a
94 coupon with payment date \p payDate. If \p alwaysAddIfPaysOnSettlement is true the fixing date will be added
95 if the coupon pays on the settlement date even if the cashflow returns hasOccured(settlementDate) as true. This
96 is conservative and necessary in some cases since some pricing engines in QL (e.g. CapFloor) do not respect
97 hasOccured() and ask for the fixing regardless. If the payDate is not given, it defaults to Date::maxDate()
98 meaning that the added fixing is relevant unconditional on a pay date */
99 void addFixingDate(const QuantLib::Date& fixingDate, const std::string& indexName,
100 const QuantLib::Date& payDate = Date::maxDate(), const bool alwaysAddIfPaysOnSettlement = false);
101
102 /*! adds a vector of fixings dates \p fixingDates for an index given by is ORE index name \p indexName arising from
103 a coupon with payment date \p payDate */
104 void addFixingDates(const std::vector<QuantLib::Date>& fixingDates, const std::string& indexName,
105 const QuantLib::Date& payDate = Date::maxDate(),
106 const bool alwaysAddIfPaysOnSettlement = false);
107
108 /*! add a single fixing date \p fixingDate for a coupon based on a zero inflation index given by its ORE index name
109 \p indexName with payment date \p payDate */
110 void addZeroInflationFixingDate(const QuantLib::Date& fixingDate, const std::string& indexName,
111 const bool indexInterpolated, const Frequency indexFrequency,
112 const Period& indexAvailabilityLag,
113 const CPI::InterpolationType coupopnInterpolation, const Frequency couponFrequency,
114 const QuantLib::Date& payDate = Date::maxDate(),
115 const bool alwaysAddIfPaysOnSettlement = false);
116
117 /*! add a single fixing date \p fixingDate for a coupon based on a yoy inflation index given by its ORE index name
118 \p indexName with payment date \p payDate */
119 void addYoYInflationFixingDate(const QuantLib::Date& fixingDate, const std::string& indexName,
120 const bool indexInterpolated, const Frequency indexFrequency,
121 const Period& indexAvailabilityLag, const QuantLib::Date& payDate = Date::maxDate(),
122 const bool alwaysAddIfPaysOnSettlement = false);
123
124 /*! clear all data */
125 void clear();
126
127 /*! add data from another RequiredFixings instance */
128 void addData(const RequiredFixings& requiredFixings);
129
130 /*! Set all pay dates to Date::maxDate(), fixingDatesIndices() will then not filter the required fixings by the
131 given settlement date any more. Needed by total return swaps on bonds for example, where a cashflow in a bond with
132 past payment date can still be relevant for the payment of the current return period. */
133 void unsetPayDates();
134
135private:
136 // FixingEntry = indexName, fixingDate, payDate, alwaysAddIfPaysOnSettlement
137 using FixingEntry = std::tuple<std::string, QuantLib::Date, QuantLib::Date, bool>;
138 // InflationFixingEntry = FixingEntry, indexInterpolated, indexFrequency, indexAvailabilityLag
139 using InflationFixingEntry = std::tuple<FixingEntry, bool, Frequency, Period>;
140 // ZeroInflationFixingEntry = InflationFixingEntry, couponInterpolation, couponFrequency
141 using ZeroInflationFixingEntry = std::tuple<InflationFixingEntry, CPI::InterpolationType, Frequency>;
142
143 // maps an ORE index name to a pair (fixing date, pay date, alwaysAddIfPaysOnSettlment)
144 std::set<FixingEntry> fixingDates_;
145 // same as above, but for zero inflation index based coupons
146 std::set<ZeroInflationFixingEntry> zeroInflationFixingDates_;
147 // same as above, but for yoy inflation index based coupons
148 std::set<InflationFixingEntry> yoyInflationFixingDates_;
149
150 // grant access to stream output operator
151 friend std::ostream& operator<<(std::ostream&, const RequiredFixings&);
152};

240/*! Inflation fixings are generally available on a monthly, or coarser, frequency. When a portfolio is asked for its
241 fixings, and it contains inflation fixings, ORE will by convention put the fixing date as the 1st day of the
242 applicable inflation period. Some market data providers by convention supply the inflation fixings with the date
243 as the last date of the applicable inflation period. This function scans the \p fixings map, and moves any
244 inflation fixing dates from the 1st day of the inflation period to the last day of the inflation period. The key
245 in the \p fixings map is the index name and the value is the set of dates for which we require the fixings.
246
247 If inflation indices have been set up via ZeroInflationIndex entries in the Conventions, the \p conventions
248 should be passed here. If not, the default \c nullptr parameter will be sufficient.
249*/
250void amendInflationFixingDates(std::map<std::string, std::set<QuantLib::Date>>& fixings);
251
252/*! Add index and fixing date pairs to \p fixings that will be potentially needed to build a TodaysMarket.
253
254 These additional index and fixing date pairs are found by scanning the \p mktParams and:
255 - for MarketObject::IndexCurve, take the ibor index name and add the dates for each weekday between settlement
256 date minus \p iborLookback period or \p oisLookback period and settlement date. The distinction between
257 Ibor and OIS is made here to cover the fixings necessary for OIS futures. The default value of 4 months
258 covers OIS futures with a contract period of up to 3 months. It would need to be configured differently
259 if OIS futures with a longer contract period are possible.
260 - for MarketObject::ZeroInflationCurve, take the inflation index and add the first of each month between
261 settlement date minus \p inflationLookback period and settlement date
262 - for MarketObject::YoYInflationCurve, take the inflation index and add the first of each month between
263 settlement date minus \p inflationLookback period and settlement date
264 - for MarketObject::CommodityCurve, add \e fixings for future contracts expiring 2 months either side of the
265 settlement date. The fixing dates are added for each weekday going back to the first day of the month that
266 precedes the settlement date by 2 months. The approach here will give rise to some spot commodities being
267 given a future contract name and dates added against them - this should not be a problem as there will be
268 no fixings found for them in any case.
269
270 The original \p fixings map may be empty.
271*/
272void addMarketFixingDates(std::map<std::string, std::set<QuantLib::Date>>& fixings,
273 const TodaysMarketParameters& mktParams,
274 const QuantLib::Period& iborLookback = 5 * QuantLib::Days,
275 const QuantLib::Period& oisLookback = 4 * QuantLib::Months,
276 const QuantLib::Period& bmaLookback = 2 * QuantLib::Weeks,
277 const QuantLib::Period& inflationLookback = 1 * QuantLib::Years,
278 const std::string& configuration = Market::defaultConfiguration);
